# Loop Builder (Ralph for Beads)

## Overview

A beads-native Ralph loop that works directly off the bead DAG instead of markdown plans. The inner loop queries `bd ready` for the next unblocked bead within a configured scope, reads the bead's description as its work instructions, executes the work, and closes the bead. The outer loop runs N iterations or until no ready beads remain in scope.

Beads are the single source of truth — the plan, the progress tracker, and the dependency graph. No separate progress files or plan markdown needed.

## How It Differs From Existing Ralph

| Aspect | Traditional Ralph | Beads Ralph |
|--------|------------------|-------------|
| Plan source | Markdown plan file | Bead descriptions in the DAG |
| Progress tracking | `.ralph/progress.json` or `.md` | Bead status (open → in_progress → closed) |
| Next step selection | Agent reads plan, picks next | `bd ready` with filters |
| Dependencies | Implicit in plan ordering | Explicit in bead DAG via `bd dep` |
| Completion signal | `RALPH_DONE` string in output | `bd ready` returns nothing in scope |

## Inner Loop (`ralph-once.sh`)

```
1. Query: bd ready --json, filtered by --slices and --pool
2. If no ready beads in scope → exit with RALPH_DONE
3. Pick the first ready bead (or highest priority)
4. bd update <id> --status=in_progress
5. Read bead description → this IS the work instructions
6. Invoke claude/gemini with a prompt that includes:
   - Project context (CLAUDE.md / AGENTS.md)
   - The bead's full description as the task
   - The agent pool constraints (which model, what tools)
   - Instructions to commit work and report results
7. On success: bd close <id> with notes summarizing what was done
8. On failure: bd update <id> --notes="failed: <reason>", leave open
9. Git commit any changes
```

## Outer Loop (`ralph.sh`)

```
for i in 1..N:
  ralph-once.sh --slices $SLICES --pool $POOL
  if output contains RALPH_DONE → break
done
```

Logs to `.ralph/logs/ralph-<timestamp>.log` via tee.

## Filters

### Slice filter
Limits which slices Ralph works on. Accepts:
- Explicit list: `--slices "auth,camera-list,map-view"`
- Slice type: `--slice-type vertical` or `--slice-type horizontal`
- All: omit flag to work on any ready bead

### Agent pool filter
Limits which pipeline steps Ralph picks up, based on the `agent_pool` label on the bead:
- `--pool test-author` — only test design, test implement, test verify steps
- `--pool code-author` — only implement, verify steps
- `--pool general` — only research, plan, report steps
- Omit to work on any step (requires the model to handle all step types)

### How filters work
The script queries `bd ready --json`, then filters the results by:
1. Bead's parent molecule matches a slice in the `--slices` list
2. Bead's `agent_pool` label matches `--pool`

First bead that passes both filters gets picked.

## Prompt Template

The inner loop constructs a prompt from:

```
You are working on a mobile app migration project.

## Your Task

{bead title}

## Instructions

{bead description — contains the full plan for this step}

## Context

- Slice: {slice name} ({slice type})
- Step: {step name} (e.g., "test-design", "implement")
- Files in this slice: {file list from slice data}

## Rules

- Read the bead description carefully — it contains everything you need
- Commit your work with a descriptive message
- When done, output RALPH_STEP_DONE
- If you are blocked or stuck, output RALPH_BLOCKED with a reason
```

## Bead Description Content

For the loop to work, bead descriptions must contain actionable instructions, not just labels. The bead scaffolder (formula + pour) must populate descriptions from the slice planning outputs:

| Step | Description contains |
|------|---------------------|
| Research | Slice file list, what to analyze, what questions to answer |
| Plan | Research findings reference, what to plan, output format |
| Test Design | Slice behavior summary, GWT format instructions, coverage expectations |
| Test Implement | Test design reference, Maestro conventions, flow directory |
| Test Verify | Flow locations, how to run maestro, debug/retry instructions |
| Implement | Plan reference, test flows to satisfy, coding conventions |
| Verify | How to run tests against new app, parity check instructions |
| Report | Artifact locations, report format, where to write output |

This content is generated by the slice planning skills and baked into the formula variables at pour time.

## Implementation Checklist

### Phase 1: Inner loop script
- [ ] `scripts/ralph-once.sh` — queries `bd ready`, filters by slices and pool
- [ ] Picks first matching bead, marks in_progress
- [ ] Reads bead description and constructs prompt from template
- [ ] Invokes claude with appropriate permission mode and allowed tools
- [ ] On exit: closes bead on success, adds failure notes on failure
- [ ] Exits with RALPH_DONE if no ready beads in scope
- [ ] Verify: run once against a test molecule with a trivial bead, confirm it picks up, works, and closes

### Phase 2: Outer loop script
- [ ] `scripts/ralph.sh` — accepts `--slices`, `--pool`, `--iterations` args
- [ ] Runs inner loop in a for loop with early exit on RALPH_DONE
- [ ] Logs to `.ralph/logs/ralph-<timestamp>.log`
- [ ] Verify: run with 3 iterations against a molecule with 2 ready beads, confirm it does 2 and exits

### Phase 3: Multi-model support
- [ ] `--model` flag to select claude vs gemini (or specific model)
- [ ] Pool-to-model mapping config file (`.ralph/models.yaml`)
  ```yaml
  test-author: codex  # or claude, determined by experimentation
  code-author: claude  # or codex
  general: claude
  ```
- [ ] Inner loop reads model from config based on bead's pool label
- [ ] Verify: run with different pools, confirm correct model is invoked

### Phase 4: Rich bead descriptions
- [ ] Update `mol-slice-pipeline` formula to include detailed description templates per step
- [ ] Description templates reference slice variables (name, files, type)
- [ ] Pour script populates descriptions from slice planning skill outputs
- [ ] Verify: pour a molecule, read bead descriptions, confirm they contain actionable instructions

### Phase 5: Safety and observability
- [ ] Max consecutive failures before stopping (avoid burning tokens on a stuck bead)
- [ ] Bead-level time limit (kill inner loop if it runs too long on one bead)
- [ ] Summary at end of outer loop: beads completed, beads failed, beads remaining in scope
- [ ] `bd ready` count check before starting (warn if scope has 0 ready beads)
- [ ] Verify: simulate a failure, confirm loop handles it and continues to next bead

## Out of Scope
- Parallelizing multiple Ralph loops (future: multiple loops with non-overlapping slice scopes)
- Automatic scope selection (future: builder recommends which slices to batch)
- Container-based execution (can be added later following existing docker pattern)
